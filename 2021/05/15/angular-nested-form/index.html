Angular nested forms with ControlValueAccessor interface

This post has but one message for Angular users: Your nested form component should implement a ControlValueAccessor.

I am not the first one to do this. Nevertheless a disclaimer and a request: I’ve had about four weeks of professional Angular development experience, apply healthy skepticism with abundance. I’m writing to clarify my understanding, and give something to the community. If you think the claim is preposterous, you know where to find me ;)

Now, let’s get you convinced by building up a minimally realistic example.

Suppose we have a form that our users use to specify how many customers we can accept in a given day. Initially, the form is very simple. There is one integer input control per day. The only validation requirement is that the capacity can’t be smaller than 0 or larger than 20.

We implement this with one component that has a FormGroup member/propery and bind the controls manually. This is the simplest use of Angular Reactive Forms. You can find a barebones example here.

Now let’s imagine a more involved second iteration for the form. We need to be able to support varying capacity throughout the day. Instead of a single integer input, we allow the user to add/remove time periods, which are specified by a start time, an end time and the capacity. We need to make sure that user can’t enter an end time that’s earlier than start time. And we need to make sure that the total capacity input doesn’t exceed 20 within a day.

We have two conceptual levels to our input: the time period level and the day level. We can implement two components, and nest the time period component in the day component. We can then reuse the time period component within and across days, and the day component across days.

I think there are three key implementation details worth noting:

  1. Components that implement ControlValueAccessor interface bind to FormControls [Eliseo’s SO answer], and not to a FormGroup or FormArray.
  2. You need to notify parents of value changes, just like the default value accessors for e.g. number inputs do.

ControlValueAccessor interface is very helpful here, for instance while handling nested arrays that do not have a predetermined size. While using it, make sure you bind it to a FormControl. Hint: The interface is called a ControlValueAccessor for a reason ;) This brings us to the third implementation detail:

  1. You need to trigger change detection if a parent acts in a way that a child causes a change in the validity of the parent without triggering change detection automatically. This is the case e.g. when the parent triggers initialisation of a child which initializes in invalid state.

In my limited experience, this implementation is the most flexible and the most idiomatic one. Do you disagree? I’d be happy to learn from you!
